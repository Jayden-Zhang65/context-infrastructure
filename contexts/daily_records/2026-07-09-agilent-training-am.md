## 四、系统网络架构

### 物理拓扑

```
用户终端（域控 Win 账号）
        │
        ▼
   ┌─────────────────┐
   │  数据服务器       │  ← 赛默飞提供虚机（超融合平台）
   │  （所有数据存此）  │     自带每日全备（0 点）
   │  含：原始数据、    │     备份目标：另一台独立机器
   │  结果集、方法、    │
   │  权限配置、活动日志│
   └─────────────────┘
        │
        ▼
   ┌─────────────────┐
   │  AIC（仪器控制器） │  ← 林工做的虚机（在客户超融合上）
   │  1 台              │     可挂 6 个仪器引擎
   └─────────────────┘
        │ (独立交换机)
        ▼
   ┌─────────────────┐
   │  仪器专用交换机    │  ← 小交换机，仅连仪器和 AIC
   │  （与办公网隔离）  │     线缆颜色区分：上排连仪器 / 下排连电脑
   └─────────────────┘
        │
   ┌─────┴─────┐
   ▼           ▼
  仪器 1      仪器 2
 (DAD)       (DAD)
```

### IT 运维要点

| 项目 | 详情 |
|------|------|
| **AIC 容量** | 1 个 AIC 可挂 6 个仪器引擎。当前 2 台 DAD = 占 4 个槽位，剩 2 个空槽。⚠️ **不建议满配**（6 个全用会满负荷），扩容需加 AIC |
| **域控** | 客户现有域控账号直接登录，无需新建。域控与 OpenLab 分开管控，域控故障不影响 OpenLab 登录 |
| **网络隔离** | 仪器和 AIC 通过**独立小交换机**连接，与办公网物理隔离。线缆分色标识（上排仪器 / 下排电脑） |
| **物理排查** | 断线时检查：线缆 → 交换机指示灯 → AIC。仪器掉线在活动日志中表现为"四个模块同时通信错误" |
| **大交换机** | 赛默飞仪器和安捷伦客户端都接在上面。这个多线的交换机管赛默飞 + 安捷伦电脑端 |

## 五、AIC 运维（IT 重点关注）

### 5.1 定期重启

| 项目 | 建议值 |
|------|--------|
| 频率 | **每 2–3 周一次** |
| 推荐时间 | **周一早上**（周末序列跑完，新任务未开始） |
| 备选时间 | 周五下午（但有序列可能在跑的风险） |
| 操作方式 | 远程进 AIC 界面 → 重启 |
| 原因 | 长期运行积累缓存/临时文件/报错状态，重启清理后可恢复性能 |
| 制度化 | **写入 SOP**，作为定期维护任务执行 |

> 📝 **批注**：这是典型的 Windows 服务器运维最佳实践——定期重启清理内存碎片和僵尸进程。建议在 SOP 中增加重启后验证步骤：确认 AIC 服务自启动成功、两个仪器连接恢复、序列可正常提交。

### 5.2 仪器连接假死处理（实验室可自主操作）

**现象**：ping 仪器 IP 通，但软件中连接仪器失败（"连接不到仪器"）。

**根因**：AIC 与仪器建立的采集进程因网络瞬时抖动（走动碰到线缆、交换机微断等）进入假死状态——物理链路自恢复但进程已僵死。

**处理步骤**：
1. **先 ping 仪器 IP** → 判断是物理不通还是进程假死
   - 不通 → 查线缆/交换机，IT 介入
   - 通 → 假死，执行以下步骤
2. 在 AIC 上打开**任务管理器** → 找到对应仪器的采集进程（名称为 `AQ` / `Server`）
3. 根据仪器的系统 ID 匹配确认后，**结束该进程**
4. 在客户端重新连接仪器 → 自动重建连接

> 📝 **批注**：建议 IT 预写一个批处理脚本（`kill_aq_by_instrument_id.bat`），实验室人员只需双击运行。脚本逻辑：`taskkill /PID <对应进程>`。降低实验室对 IT 的依赖。
