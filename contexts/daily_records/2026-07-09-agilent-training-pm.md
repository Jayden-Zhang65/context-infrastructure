## 二、项目结构体系

### 2.1 层级模型

```
Project Group（项目组）
  └── Project（项目）
        ├── Methods（方法）
        ├── Sequences（序列表）
        ├── Results（结果集）
        ├── Report Templates（报告模板）
        └── Data（原始数据）
```

- **Project Group**：纯组织容器（"空壳套娃"），不存数据，仅用于归类
- **Project**：真正数据容器，创建时自动生成上述 5 个子目录
- 层级可无限嵌套 Project Group，但**最后一级必须是 Project**
- ⚠️ **项目创建时必须勾选"在项目组路径下创建"**，否则所有数据丢在根目录

### 2.2 方法隔离机制（重点）

| 特性 | 行为 | 影响 |
|------|------|------|
| 方法归属 | 方法锁定在创建它的项目内 | A 项目的方法不能直接被 B 项目调用 |
| 跨项目复制 | 通过 Web 界面右键 → 复制到其他项目 | 会产生方法副本，各自独立演进 |
| 年度归档 | 可以按年建 Project Group，年底把方法拷到新年 | 维护成本高，方法版本容易混乱 |

> 💡 **培训师建议**：不要过度按年/月拆分项目。方法跨项目不共享是个设计约束，项目拆分越细，方法维护越痛苦。平衡点：按**产品线或方法族**分组，而非按时间。

### 2.3 IT 运维要点

- **项目命名**：创建后**绝不能重命名**。重命名会导致路径错乱，数据无法找回，系统可能报错。错了就新建一个，旧的废弃。
- **路径固化**：5 个子目录名称和路径一旦创建无法修改。只能在创建时一次做对。
- **防错机制**：建议设置双人复查流程——第一个人创建项目但不点确认，第二个人核对无误后再确认。利用电子签名功能实现。

## 三、报告模板管理

### 3.1 两种策略对比

| 策略 | 做法 | 优点 | 风险 |
|------|------|------|------|
| **A. 共享模板仓库** | 建一个专门的"报告模板"项目，所有模板放这里。其他项目通过"浏览"引用 | 统一管理，修改一处全局生效 | ⚠️ **高危**：模板被改后，历史报告重打会变样。审计/专家检查时"打出来的跟当年不一样"→ 合规事故 |
| **B. 每项目独立副本** | 每个项目自己的 Report Templates 目录下放独立模板副本 | **安全**。改模板不影响历史项目 | 模板多了维护麻烦，但 OpenLab CDS 2.8 支持从模板项目创建项目模板（rubric），可降低初始配置成本 |

> 🏆 **培训师强烈推荐策略 B**。原因：一旦模板被修改，历史版本无法追溯，"你跟专家解释不了为什么两次打出来的报告不一样"。这在 GxP 环境下是 **数据完整性问题**。

### 3.2 报告模板的版本控制风险

- OpenLab CDS 本身**不保留报告模板的历史版本**
- 模板被修改后，再对旧数据运行报告，输出结果会变
- 如果必须共用模板，**每次修改必须另存为新模板**（改名），旧模板保留不动

### 3.3 IT 运维要点

- 权限控制：限制能修改报告模板的用户（建议仅 1–2 人）
- 模板备份：每次重大修改前手动备份模板文件
- 合规建议：在 SOP 中明确"报告模板修改须经变更控制，旧版本归档保留"

## 六、物理网络与仪器排障

### 6.1 布线说明

```
仪器专用交换机（小交换机，仅几个口）
  ├── 上排 RJ45 → 仪器 1（Agilent LC）
  ├── 上排 RJ45 → 仪器 2（Agilent LC）
  ├── 下排 RJ45 → AIC
  └── 绿色跳线 → AIC ↔ 交换机之间
```

- **线缆颜色区分**：上排（连仪器）用专用线，下排（连电脑）用不同线
- **交换机端口无绑定**：随意插拔，自动识别
- **物理诊断顺序**：看交换机指示灯 → 看 AIC → Ping 仪器 IP

### 6.2 仪器掉线诊断（IT 必读）

**现象 1：四个模块同时报通信错误**

→ 肯定是物理连接问题（非软件故障）。检查交换机、线缆、AIC。

**现象 2：Ping 通但软件连不上（假死）**

```
根因：AIC 与仪器间的采集进程因网络微断（线缆松动、交换机瞬时抖动）进入僵死状态。
      物理链路自愈后进程未恢复。

处理（IT / 实验室皆可操作）：
  1. Ping 仪器 IP → 确认物理链路正常
  2. AIC 上打开任务管理器 → 找到 AQ/Server 进程
  3. 根据仪器系统 ID 匹配确认后 → 结束该进程
  4. 客户端重新连接仪器 → 自动重建

  注：不需要重启 AIC！如果两台仪器都在跑，只需 Kill 假死的那台。
```

**现象 3：两台仪器同时掉线**

→ 检查 AIC 本身（死机/蓝屏/重启）或交换机。AIC 是用久了的旧电脑，存在硬件故障可能。

**现象 4：偶尔单个仪器掉线但自恢复**

→ 大概率是某根网线接触不良。做法：记录掉线时段 → 换一根线 → 观察是否复现 → 如果继续掉换另一根 → 二分定位。

### 6.3 IT 运维要点

- **交换机是林工从赛默飞报的一台**，不是工业级交换机，如果有频繁掉线考虑升级
- **建议写一个 ping 监控脚本**（ping 仪器 IP 每 5 分钟一次，不通则报警），提前发现物理链路问题
- **任务管理器 Kill 进程的操作**可以教给实验室班长，减少 IT 响应压力

## 八、网络安全

### 8.1 讨论要点

| 议题 | 讨论结论 |
|------|----------|
| **仪器网段连外网风险** | 38 网段当前连着外网 → 有中勒索病毒先例（培训师前两个月刚处理过一例）。**强烈建议断外网**，或至少将 38 网段切到纯内网 |
| **U 盘管控** | 安捷伦有计划铺开安全软件（试点 10 台），限制 U 盘插入和拷出。当前过渡方案：在 101 电脑开只读共享，数据只能往外读、不能往里写 |
| **共享文件夹管理** | IT 管理员在服务器上建了共享（`\\server\share`），用于数据拷出。但需要管理员权限才能访问 |
| **域控独立性** | 域控账号与 OpenLab 账号分开管理，域控故障不影响 OpenLab 登录 |

### 8.2 IT 运维要点

- **38 网段断外网**是优先级最高的安全事项，建议尽快执行
- **U 盘管控**正式上线前，做好用户沟通（实验室人员可能依赖 U 盘传数据）
- **只读共享**是一种低成本过渡方案，但需要明确 SOP：谁可以放文件到共享、多久清理一次
