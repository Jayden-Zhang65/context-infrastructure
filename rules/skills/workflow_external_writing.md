# 外部写作工作流

## 元数据

- **类型**: Workflow
- **适用场景**: 将调研素材转化为有判断力的 external-facing 分析文章、公开 survey report 或面向外部读者的课程/客户内容
- **前置依赖**: 深度调研工作流（`workflow_deep_research_survey.md`）的 Phase 1-3 产出（或等价的已验证素材）
- **创建日期**: 2026-04-28
- **最后更新**: 2026-06-17

## 这个 skill 解决什么问题

调研素材充分但写出来像 AI 汇总，缺乏作者的分析视角和判断框架。本 skill 把"一堆验证过的事实"变成"一篇陌生读者能理解、记住并复述的分析"。

**适用边界**：面向没有共享上下文的陌生读者、公开发布渠道、客户或外部课程受众。面向鸭哥本人、共享上下文协作者或未来 AI agent 的内部文档用 `workflow_internal_writing.md`。

**产出类型判断（动笔前必须先做）**：external-facing 调研分析的**默认产出是 survey report**，存 `contexts/survey_sessions/`，不加 Pelican frontmatter，不加 Kit 订阅脚本。只有当用户明确说"写博客"、"发到博客"、"blog post"时，才按博客处理，存 `contexts/blog/content/` 并加 frontmatter。"写一篇文章"不等于 blog，默认走 survey report 路径。判断错误会污染 frontmatter、错放存储位置、带偏配图策略。

**与深度调研 skill 的关系**：深度调研 skill 管信息采集和验证，本 skill 管判断合成和写作。调研完成后指向这里。

## 验收标准：好的 external 文章长什么样

写完后用以下标准自查。这些是结果定义，不是过程步骤。

**第 1 条是门控约束，不满足直接退回重写，其他条目都是在它满足之后才讨论的优化。**

1. **非专家能看懂并复述 thesis**：找一个不在这个领域工作的人，把文章念给他听。他能不能在听完后用一两句话复述文章在说什么？如果他在某一节开始皱眉或走神，那一节的信息密度或背景假设出了问题。这是所有标准里最硬的一条——不是"写得顺不顺"，是"读者到底有没有看懂"。AI 写作最常见的失败不是句式问题，是默认读者已经知道 SFT、RLHF、零信任、债券利差这些概念，然后在读者还没消化上一个概念时就推到下一个。**这条不满足，下面的条目都不用查。**

2. **有 thesis**：文章有一个核心判断，一个不了解细节的聪明人读完能用 1-2 句话复述。不是"这件事很复杂值得关注"，而是具体的、可被同意或反对的判断。

3. **Thesis 前置**：文章的前 25% 读完，读者知道 (a) 这篇文章在讲什么问题、(b) 作者的判断是什么、(c) 继续读能得到什么。三个问题任何一个需要读到后半篇才能回答，结构需要调整。

4. **围绕读者组织，不是围绕作者的分析框架组织**：文章按读者的疑问链推进（震惊 → 恐惧/好奇 → 释然 → 现实），不按作者的分析维度（"组合爆炸 + 绕过"这类分类）罗列。判断标准：把文章读给目标读者听，他会不会在某一段想"这关我屁事"。

5. **证据经济**：每个论点用一个最强案例，不堆砌同类证据。读者在第一个案例之后就在想"所以呢"，如果接下来还是同类证据，注意力已经用完。

6. **文字干净**：通过 post-writing 扫描（破折号、"很+形容词"判断总结、内部框架术语泄漏、大段英文引文）。详见 `rules/COMMUNICATION.md`。

7. **有信息量的小标题**：超过 1500 字的文章用 `##` 级小标题。小标题携带信息不做范畴名——删掉后正文第一句话能否独立充当小标题？能，说明小标题只是范畴名。

8. **分析框架不可见**：Thesis Catalog（L1-L8）、axiom 编号、Phase 名称、"叙事重构"等术语属于写作过程的脚手架，不能出现在最终文章里。读者感受到的是一个有自己思考方式的人在做分析。

## Thesis Catalog：启发性分析视角

以下视角是从作者已发表文章中归纳出的分析习惯，不是必须套用的框架。它们有两个用途：一是帮你找到文章的切入点（"这件事的底层机制是什么""大众忽略了什么维度"），二是让文章带有作者的判断风格，而不是 AI 汇总的味道。

用多少、用哪些完全由文章决定。很多好文章一个视角都没显式用到——作者本来就有自己的判断。把这些视角当灵感库而非清单：如果某个视角和当前话题自然契合，用它；如果不契合，忽略它。不要为了"用上某个 L"而强行套框架。视角指导你怎么想，但读者看到的只有想出来的结果。

L1-L3 有一个共同主题：诊断 AI 系统的局限来自哪里（L1 诊断反馈环、L2 诊断瓶颈和成本结构、L3 诊断训练先验）。它们经常在同一篇文章里互补使用。

### L1: 反馈闭环判断

一个系统的能力天花板取决于它能否感知自身输出并自我修正。AI 系统中最关键的不是模型智能，而是模型能否看到自己行为的后果。闭环完整的系统可以自我迭代，闭环缺失的系统只能依赖人类补位。这个原理不只适用于系统设计，也定义了学习的本质：持续校准方向的能力比一次瞄准的精确更重要。

**对应 axioms**: A02（放大器）、A04（可靠性是管理问题）、M01（闭环校准）

**典型应用**: 创意 AI 三代演化（Generation 1 无法看到渲染结果，Generation 3 通过截图闭环）、Agentic AI 部署危机（代码能跑但不知道对不对）、闭环学习（开环 vs 闭环控制论应用到个人学习哲学）

**触发条件**: 分析对象涉及 AI 系统的可靠性、自动化程度、或"为什么 demo 很好但实际用不了"

### L2: 成本结构经济学重建

当某项关键资源的成本发生数量级变化时，系统性地重建旧经济学和新经济学：旧的最优策略在旧成本下为什么合理，新的最优策略如何不同。不只是说"瓶颈迁移了"，而是完整对比两套经济学。关键洞察：我们引以为豪的很多实践（DRY 原则、直觉决策、精心设计防御性代码）不是天生正确的，而是旧成本结构下的合理产物——在新成本结构下它们可能变成负担。

**对应 axioms**: X03（效率由瓶颈决定）、T05（认知是资产，代码是消耗品）

**典型应用**: AI Native 成本结构（代码贵→复用/DRY vs 代码趋零→一次性/用完即焚）、过程确定性到结果确定性（人力贵→精心设计 vs intelligence 便宜→挥霍 token 换确定性）、Context infrastructure（模型智能是瓶颈→升级模型 vs context 是瓶颈→积累 context）

**触发条件**: 分析对象声称"让 X 变得更快/更便宜/更容易"时，追问旧的最优策略为什么合理、新的最优策略如何不同

### L3: 共识天花板判断

LLM 的默认输出是共识（consensus）。训练机制（next token prediction + RLHF 安全对齐）决定了 LLM 回归均值。差异化和深度来自注入非共识的个人上下文，不来自换更好的模型。Deep Research 其实是 Wide Research。主要适用于内容创作和分析质量类话题。

**对应 axioms**: context infrastructure 博客的核心论点、A10（熟悉度胜过原始智能）

**典型应用**: Context infrastructure（两份报告的对比实验）、对 Deep Research 产品的批判

**触发条件**: 分析对象涉及"AI 能不能做 X"的能力边界讨论，或者任何 AI 产出质量的评估

### L4: 技术族谱追溯

任何新发布都不是凭空出现的。追溯它的前几代可以定位它在演化轨迹上的真实位置，区分"真正的新东西"和"已有路径的产品化"。通常分成 2-4 代，每代解决了前一代留下的核心问题。主要适用于产品评测和技术发布分析。

**对应 axioms**: M06（连接胜过孤立知识）、A13（技术采用三阶段）

**典型应用**: 创意 AI 三代（脚本 → 协议 → 闭环）、MCP 演化（研究协议 → 工程现实碰撞 → 方言分裂）、GPT-5 定位（o3/Gemini/Claude 的"个性"被参数化统一）

**触发条件**: 分析对象是一个新发布的产品/技术/标准，需要定位它在演化中的位置

**操作方法**: 第一步，识别前几代：这个东西要解决什么问题？在它之前人们怎么解决？前一代的核心局限是什么？第二步，画出代际演化线：每代解决了什么、留下了什么。第三步，定位当前事件在演化线上的位置：它解决了哪一代遗留的问题？它本身又会遗留什么？第四步，判断演化的下一步：基于当前遗留问题，下一代可能长什么样？

### L5: 表层诊断与底层机制

表面看到的不是真正起作用的东西。文章的核心工作是把注意力从表层重定向到底层机制。这个"表层"可以是媒体/厂商叙事（"AI 能做设计了"）、拟人化直觉（"AI 偷懒了"）、或成功场景里藏着的隐含前提（"demo 能跑 = 达到目标"）。共同结构是：人们停在表层描述上做判断，而真正驱动结果的是底层机制。

**对应 axioms**: T04（数据优于观点）、V02（可验证性是信任的地基）

**典型应用**: 几乎每篇 external 文章都有这个动作。创意 AI（叙事聚焦"AI 能做设计了"，实际瓶颈是反馈闭环）、Wide Research（"偷懒"掩盖了架构性约束）、AI Native 成本结构（"直觉是高手象征"实际是观测成本高时的被迫压缩策略）

**触发条件**: 几乎总是需要。问自己：关于这个话题，大多数人的第一反应是什么？这个第一反应是在看表层还是在看机制？拟人化描述掩盖了什么技术本质？

**操作方法**: 第一步，提取主流叙事或表面判断。第二步，识别它把注意力放在了哪个维度。第三步，追问底层机制：真正驱动结果的是什么？第四步，构造重定向。不是每篇都需要"反驳"主流叙事，有时是"推深一层"。

### L6: 执行摩擦消除后的价值重分配

当工具消除了执行层面的摩擦，竞争优势从"谁执行得快"转移到"谁的判断更好"。工具掌握的价值下降，品味和方向感的价值上升。在具体文章中，这个判断通常以角色重定位的形式出现——不只是抽象地说"判断更重要"，而是描绘出人类在新系统中的具体角色。三种常见变体：IC→Manager（管理类，从亲力亲为到设计流程让系统自我迭代）、User→Builder（工具使用类，从被动调用到主动构建和改进工具）、出题人 vs 执行者（工作流类，人的核心价值从"做"转移到"定义做什么、判断做得好不好"）。

**对应 axioms**: T05（认知是资产）、A02（AI 是放大器不是替代品）

**典型应用**: 创意 AI（人类从创作者转向边界设计者/反馈解释者）、Wide Research（Senior Manager 视角设计流程）、Deep Research（AI 数据挖掘 + 人类认知炼金的分工）

**触发条件**: 分析对象涉及"AI 让某类工作变简单了"时，追问价值转移到了哪里，以及人类的新角色具体长什么样

### L7: 管理框架平移

AI 使用中遇到的问题往往不是 AI 特有的新问题，而是人类组织管理中早已解决的问题。把 AI 当团队成员来管理，可以调用成熟的管理框架（招聘/委托/培训/指导/验收）获得现成解法。"太阳下没有新鲜事"——AI 的不可靠、幻觉、偷懒、沟通成本，在人类组织中都有对应的经典管理问题和解决方案。这个视角的力量在于：它把"AI 是全新的、需要全新方法论"的焦虑，转化为"我们其实已经知道怎么做"的务实感。

**对应 axioms**: A03（IC→Manager）、A04（可靠性是管理问题）

**典型应用**: AI 管理三部曲（不抢键盘/可视化上下文/授之以渔，每一条都映射到经典管理学）、AI 关键决策（manager 设计流程让 AI 自我迭代，而非跟进每个 work item）、AI 管理 2（"AI 不可靠"映射到"新人不可靠"，解法是情境领导力和层级化扩展）

**触发条件**: 分析对象涉及 AI 使用中的可靠性、效率、协作质量问题，且这些问题在人类组织管理中有成熟对应物

### L8: 上下文与信息架构

系统的能力边界和故障模式取决于它"能看到什么"——上下文里有哪些信息、信息是否整洁、不同角色的信息是否隔离。分析任何 LLM 系统的第一步，是逆向理解它的 context window 里到底是什么。这与 L1 互补：L1 讲"能否感知并修正"（动作层），L8 讲"感知的信息基础是什么"（信息层）。当 AI 的行为出问题时，先问"它看到了什么信息"再问"它的推理对不对"。

**对应 axioms**: M01（闭环校准）、context infrastructure 博客核心论点

**典型应用**: Multi-agent 改造（角色拆分、文档沟通、Planner 选择全部由 context window 管理倒推）、Context infrastructure（模型智能跨过门槛后 context density 成为瓶颈）、AI Software Engineering（激进式透明 = 给 AI 足够的反馈信息让它自我修正）

**触发条件**: 分析对象涉及 LLM 系统的故障模式、能力边界、或"为什么同一个模型在不同场景表现差异大"。也适用于设计 AI 系统时的信息架构决策（哪些信息进 context、哪些外化到文件、哪些隔离到不同 agent）

## 从素材到 thesis

以下是从调研素材到可写作 thesis 的可用分析工具。不是流水线——如果用户已经给了 thesis（比如来自一段讨论或一个明确观点），跳过发现过程直接进入写作。只有当 thesis 需要从素材中发现时才用这些工具。

**视角匹配**：扫描 Thesis Catalog（L1-L6），标注哪些与当前话题强相关、辅助、不相关。一篇文章通常用到 1-3 个视角。

**搜索作者已有写作**：在 `contexts/blog/`、`contexts/survey_sessions/`、`rules/axioms/` 中查找与话题直接相关的已有文章、调研和公理。如果用户在对话中提到了自己的相关思考或已有文章，这些是 thesis 的最强锚点。

**技术族谱追溯**（L4 强相关时）：识别 2-4 代前身，画出演化线，定位当前事件。

**叙事重构**（L5 强相关时）：提取主流叙事 → 识别叙事聚焦的维度 → 识别被忽略的维度 → 构造重定向。

**输出**：thesis statement（2-3 句话，能被不了解细节的聪明人理解，包含具体判断）+ 论证骨架（3-5 个要点，每个标注用了哪个视角和哪些证据）。写入 `tmp/<session_slug>/scratchpad.md`。

## 写作原则

动笔前必读 `rules/COMMUNICATION.md`。不是写完回来对照检查，是动笔前把规则装进来。以下只重复长文写作中 AI 最容易犯的几条，完整规则以 COMMUNICATION.md 为准。

### 结构：围绕读者组织

**Thesis 前置与证据经济性**。文章结构围绕读者的心理推进设计：钩子（读者能感知的现象）→ 这件事和你的关系 → 核心论点（1-2 句话）→ 证据（每个论点一个最强案例）→ 解法 → 限定。不要把多个章节花在证明同一个论点上。不要按作者的分析框架（"维度一 + 维度二"）组织，按读者的疑问链组织。判断标准：读者会不会在某一段想"这关我屁事"。

**小标题**。超过 1500 字的文章用 `##` 级小标题。小标题携带信息不做范畴名——"为什么命令行过滤失效"是范畴名，"组合爆炸：shell 语法的等价表达无穷多"携带了信息。数量 3-5 个，每个下面 2-4 段。

**标题策略**。标题是文章在信息流里的唯一入口。事件驱动型必须包含具体事件关键词（产品名、公司名、动作词）。分析型可以从判断或张力入手但仍要给出具体对象。判断标准：把标题读给一个关注 AI 但没听说这件事的人听，如果他问"这是什么"，标题就失败了。

**文章重量校准**。默认目标长度 2500-3500 字，不再分快评 / 分析 / 深度调研三档。这个长度对应：完整 Phase 调研流程、完整证据链、3-5 个论证要点配小标题、论证密度足够支撑 thesis 而不压缩成清单。配图按需，深度调研型必须配，分析型可以省。短文（1500-2000 字）只在用户显式要求时才写，并且要明确说"这一篇是短文，因为 X"；不要默认把文章往短写。"low cognitive burden" 不等于短，等于读者顺着一条自然的故事线读下去不觉得被灌了东西。判断一篇文章该多长，看论证需要的呼吸空间，不看一个固定的字数预算。**例外**：当认知负担是主要矛盾时（读者反馈"看不懂"），砍内容比改写更有效，即使砍完后短于默认长度。一篇 3000 字但读者能看懂的文章，比一篇 7000 字但读者只看懂三分之一的文章有价值得多。

### 认知负担

低认知负担这个目标本身是对的，但它在 AI 写作里被反复误解。机械地套规则——强制短句、强制短段、每两段插一次承接词、把"三件事"换成"第一件第二件第三件"——不降低认知负担，反而把一篇流畅文章切碎成 checklist。读者读到的不是文章，是被规则肢解的条目。

真正降低认知负担的做法只有一条：让读者顺着一条自然的故事线走下去，读完不觉得被灌了东西。以下几条是具体手法，来自同一篇文章反复重写到第五版才确认有效的模式。

**禁止 taxonomy 宣布。** "第一件/第二件/第三件"、"有几个方向"、"三样东西"——这类前置编号给读者的信号是"接下来是清单"，读者的注意力会从跟着走切到数到第几了。让内容自己浮现。如果三件事有先后关系，叙事本身会带出顺序；如果只是并列，那说明你在罗列而非写作。验证：删掉所有编号后文章还能独立站住吗？能，编号就是多余的报幕。

**禁止机械插承接词。** "不过"、"这样一来"、"换句话说"、"话说回来"只在阅读节奏需要呼吸时才出现。段落本身流畅时一个不加。反复写同一篇文章的多个版本最容易暴露这个问题：上一版机械插入的承接词，删掉大半后节奏反而对了。验证：出声念一遍。如果承接词像模板每两段出现一次，那是机器在填格子。如果一段念下来顺滑无顿挫，不需要硬塞。

**禁止信息墙。** 不要把五六个数据点塞进一个段落。数据在文章里的角色不是被列出，是推动叙事往前走一步。每个数据点放进去前先问：它在帮助读者理解什么？如果答案只是"它在这"，拆到自己的段落里，或合并到当前正在推进的叙事动作中。数据嵌在故事里，不是"数字→含义"的 Q&A 模式。

**段落有自然长度。** 讲一个完整小故事时该长就长，出 punchline 时该短就短。不强制任何长度。发现每段都是三到四行、节奏完全均匀时，说明在套模板。

**出声念一遍。** 写完后的最终检查不是 checklist 扫描，是把文章出声念一遍。念出来像不像人在说话？像在跟朋友讲一件有意思的事，还是像在念一份工作汇报？这个测试比任何规则都可靠——念的时候不会数"这段有几个新概念"，只会感觉到节奏对不对。

以上是叙事层的认知负担。概念层的认知负担（每段新概念不超过两个、新术语先用已有场景接住、jargon 默认不出现）仍然生效，但它们解决的是"读者能不能跟上"，叙事层解决的是"读者愿不愿读下去"。两层缺一不可。

### 降认知负担的操作方法（信息架构层）

上面讲的句式和节奏是微观层面。实际写作中，读者说"看不懂"的最常见原因不在句式，而在宏观的信息架构：一篇文章塞了太多角度、太多术语、太多案例。以下是从同一篇文章反复重写直到读者说"质的飞跃"的过程中确认有效的操作方法。

**一个 section 只放一个核心判断 + 一个具体故事。** 这是降认知负担最强的单一操作。假设你的文章有三个角度，每个角度只保留一个最能说明问题的锚点——一个公司做了某件事，或一个人说了某句话。其余的数字、引语、案例全部砍掉或压到半句话里。判断标准：读者读完这一节能不能用一句话说出"这节在讲什么"。如果他自己总结不出，说明这节塞了太多东西。

**砍掉比添加更重要。** AI（尤其 DeepSeek 等 sub-agent）的默认行为是尽可能多地使用调研素材，因为素材越多显得越"充分"。但 external-facing 文章的目标不是把调研展示给读者，是把判断传达给读者。当你发现一篇文章认知负担过高时，第一反应不应该是"怎么改写得更通俗"，而是"什么可以砍"。一篇 7000 字的文章砍到 3000 字后读者说"质的飞跃"，砍掉的不是废话，是有趣但不必要的论证、有意思但不必要的案例、准确但不必要的术语。

具体砍什么：第二个同类案例（保留最强的那个就够）、支撑性的引语（保留核心判断就够，不保留每句话的出处）、中间分析步骤（保留起点和结论就够，不保留推理链条的每一步）、术语的精确定义（用日常场景替代，不用学术定义）。

**读者已有的认知是入口。** 文章的第一段必须从读者已经知道或能直觉理解的东西开始。"OpenAI 去年花了 340 亿美元"任何人都能 grasp。"三条信号链"或"从 model safety 到 agent security 的范式转移"只有领域内的人能 grasp。前者是入口，后者不是。如果文章必须引入一个新概念，先用读者已有的场景接住它，再给名字。"互联网泡沫亏的是股东的钱，AI 泡沫亏的是债主的钱"——这是读者能直觉理解的对比，比"损失传导机制的差异"强得多。

**专业术语要么删要么翻译。** "债券利差走阔"改成"那些借钱比较吃力的公司，借钱的成本开始变高"。"单位经济学"改成"每挣一块钱要花两块六"。"搁浅资产"改成"建好了但赚不了钱的东西"。如果某个术语删掉不影响论证，直接删。如果删不掉，紧跟一句日常语言的重述。读者不需要学会你的术语，你需要学会用读者的语言说话。

### 散文规则（长文高风险项）

以下几条在长文写作中 AI 反复犯错，即使读过 COMMUNICATION.md 仍会犯，因此在此重复。完整规则以 COMMUNICATION.md 为准。

**破折号零容忍**。成品中不得出现 ——。插入补充拆成两句，因果改用句号或冒号，双破折号插入语拆成独立句。

**「很 + 形容词」判断总结**。不写"展示得很直接""说得很清楚""结果很明确"。这种句式用一个形容词跳过论证责任。写完后用 `rg '很[直接清楚简单明显重要透彻全面深入关键]'` 扫描，判断总结一律删除或替换为具体内容。

**英文引文处理**。中文文章中的英文引文按内容密度而非句号数量判断。blockquote 英文引文超过 20 个英文单词，或虽为单句但含 3 个以上关键信息点（数字、专有名词、条款）的，改为中文重述，保留 URL。短句英文（≤20 词、单一信息点）作为标志性原话可保留 blockquote。inline 短句英文（产品名、术语、≤15 词引文）可保留。中文读者面对大段英文的认知成本高：需要切换语言、解析语法、再回到中文上下文。英文版文章不受此限制。

**元评论零容忍**。不写"具体来说""这里有一个关键区别""接下来看"。直接进入内容。

**承接词**。段与段之间用轻承接（这样做、不过、换句话说、关键在于）维持节奏，不要每句都像独立砸下来的砖头。反过来也警惕机械插入：承接词太密、像固定模板每隔一段出现一次，同样破坏节奏。判断靠出声念，不靠计数。

**不用引号包裹概念词**（除非绝对有歧义）。

### 格式

- 中文 Markdown，所有引用用绝对 `https://` 链接 inline 在正文中
- 不得在文末单独列「来源」或「参考文献」清单
- 不在正文开头写发布时间、调研日期（发布时由系统注入）
- Survey report（默认产出）存 `contexts/survey_sessions/`，不加 Pelican frontmatter。博客文章存 `contexts/blog/content/` 并加 frontmatter。
- External-facing 文章配图 0-3 张。数据对比和流程图必须配，纯文字分析文章可以不配。配图放在 MD 同目录，用相对路径引用。配图默认走两步流程：先用 matplotlib 生成结构草稿（含文字标注、布局、定位），再用 gpt-image-2 以草稿为 `-i` 输入做重绘，优化视觉呈现。matplotlib-only 只适用于需要精确数据的纯定量图表（柱状图、折线图、散点图）。概念图、流程图、对比图、栈图都走两步流程。gpt-image-2 重绘 timeout 设 300s+，含中文时降低文字密度，16:9 横版，要求字体正常宽度不拉伸。详细操作见 `generate_image.md`。

## 模型分工协议

external-facing 文章的最终 prose 默认由 DeepSeek Pro 写作子 agent 完成。主线程负责调研、判断、材料整理、事实核查、风格审阅和最终验收；写作子 agent 负责把已定稿的判断写成文章。

**标准交付必须包含两遍 DeepSeek Pro 写作：第一遍成稿，第二遍语言层重写。** 第一遍负责把调研、thesis 和证据组织成完整文章；第二遍负责在不改结构、不改事实、不改信息量的前提下，用 DeepSeek Pro 自己最自然、最舒适的语言完全重写 prose。两遍合起来才算完成写作流程。除非用户明确要求只要草稿、只要 outline、或只做内部临时 memo，否则不要把第一遍成稿当作最终稿交付。

主线程不得直接起草最终 prose。无论主线程模型是什么，都必须先把调研素材、thesis、证据表、风格要求、禁用事项、目标输出路径和覆盖策略落盘到 `tmp/<session_slug>/writing_brief.md`，再调用 `writer_deepseek` 子 agent 读取该文件并写入第一遍 Markdown：`tmp/<session_slug>/draft.md`。Agent 之间不要依赖 chat 传递正文素材，文件才是交接界面。`writer_deepseek` 是 DeepSeek Pro 写作路线；不要再使用 DeepSeek V4 Flash 承担标准 external prose 写作。

主线程负责：准备材料、事实核查、风格审阅和最终验收。不把未核查的 sub-agent 初稿或重写稿直接交付。委派的是执行，问责留在自己这里。

**降认知负担的结构判断由主线程完成，语言层完全重写由 DeepSeek Pro 子 agent 完成。** 历史实测表明 DeepSeek V4 Flash 容易把重写做成同义词替换，不能承担标准 external prose 的第二遍。当前标准路线改为 `writer_deepseek`：主线程先确定文章结构、信息取舍和读者认知路径，再让 DeepSeek Pro 在这些约束内从头重写语言。若读者反馈"看不懂"，主线程先自己重组结构和删减信息，再交给 `writer_deepseek` 做语言层重写；不要把结构性判断外包给写作子 agent。

用户要求重写或覆盖最终文件时，可以覆盖 `contexts/survey_sessions/` 下的交付稿；但 `tmp/<session_slug>/` 下的 raw material（source index、scratchpad、handoff brief 等）必须保留作为审计链。

## 修订流程

收到反馈后，先诊断反馈类型再动手。

**结构问题**（"不知道你在讲什么""看完前两章没看懂""关我屁事"）：不需要改句子，需要重组文章。先确定读者疑问链（读者走到每一段时的心理状态），再围绕它重新排序。结构性反馈是最重要的信号——它说明文章的组织逻辑和读者的认知路径不匹配。

**风格问题**（"这个句子不舒服""这里有翻译腔"）：在当前位置修复，不影响结构。

**事实问题**（"这个数字不对""这个案例描述有误"）：核查来源，修正后重新验证相关论证。

## 第二遍写作：语言层重写

内容和结构定稿后、Post-writing 自查前，必须执行第二遍写作：**语言层重写**。这一轮不动结构、不动内容、不动信息量、不动 URL 和事实，只把整篇文章的 prose 从头用重写者自己的语言习惯重写一遍。

这不是可选润色，也不是发现风格问题后才做的修补。external-facing 文章的标准工作流就是：先完成第一遍成稿，再完成第二遍语言重写，最后才进入 Post-writing 扫描。第一遍成稿是 draft，第二遍重写稿才是 final candidate。

为什么要这一轮：写到这里文章的结构、thesis、证据都已经定下来，但 prose 层常常残留原作者（主线程或初稿 sub-agent）的句式惯性，包括翻译腔、节奏扁平、承接词缺失、概念词被引号包裹、"很+形容词"判断总结、破折号情绪停顿。Post-writing 扫描只能抓显性违规，抓不到节奏和语感。一个全新的写作脑，不带着初稿的惯性，能用更自然的中文重写一遍。

**执行方式**：调用 `writer_deepseek` 子 agent，把第一遍成稿 MD 路径交给它，要求它输出一个全新文件：`tmp/<session_slug>/rewrite.md`。不要覆盖原稿。主线程验收通过后，再把重写稿复制或应用到最终交付路径。

**给重写者的硬约束**：

1. **不改动结构**：章节顺序、小标题、段落数量、段落对应关系保持一致。
2. **不改动内容**：thesis、判断、案例、数字、URL、引用全部保留。
3. **不改动信息量**：每段传达的信息密度不变，不增删论点。
4. **从零新建，不是逐句修改**：你不是在原稿上改字，你是在读完后关上原稿，用自己的语言把同样的内容从头写一遍。原稿的作用是告诉你"这段要传达什么信息、引用哪些 URL"，不是给你一个模板去做同义词替换。允许换主语、换承接词、合并短句、拆分长句、调整从句顺序、重新组织段落内部的论证顺序。判断标准：把重写稿和原稿逐段对比，没有任何一段超过 30% 的句子雷同。如果超过，说明你在搬原句而不是在重写，需要推翻重来。
5. **风格遵循 COMMUNICATION.md**：重写者动笔前必读 COMMUNICATION.md，把规则装进来再动笔。

**主线程验收**：重写稿写完后，主线程逐段对比原稿，检查以下三项：(a) 结构完整性——章节顺序、小标题、段落数量一致；(b) 内容完整性——所有 thesis、判断、数字、URL 均保留；(c) 语言重写度——逐段检查句子雷同率，任何一段超过 30% 的句子雷同即判定为未完成重写，需推翻该段重来。验收通过后才让重写稿进入 Post-writing 自查环节。原稿保留作为审计链。最终回复用户时，如果没有完成第二遍重写，必须明确说明交付物仍是第一遍草稿，不能称为最终稿。

## Post-writing 质量检查

写完后必须执行以下扫描，不要靠肉眼：

```bash
# 1. 破折号
rg -n '——' <file>

# 2.「很+形容词」判断总结
rg '很[直接清楚简单明显重要透彻全面深入关键]' <file>

# 3. 内部框架术语泄漏
rg 'L[0-6]|axiom|Phase [A-Z]|叙事重构|技术族谱|瓶颈迁移|Thesis Catalog' <file>

# 4. 英文 blockquote 超长（超过 100 字符的 blockquote 行需人工检查是否超 20 词或 3 个信息点）
rg -n '^> ' <file> | awk '{ if (length($0) > 100) print }'

# 5. 引文/事实前的评价性引导（元评论变种）
rg -n '说得.{0,5}更|直白到|不需要转述|本身就说明|本身就有点|很容易漏掉|用不着在推理' <file>

# 6. 被动语态（翻译腔重灾区；中文里「被」字句低频、带强调味）
rg -n '被' <file>
```

前五个扫描必须全部空结果（#4 命中需人工判断）。#6 不会空结果，要逐个分类：哪些是翻译腔必须改写，哪些是中文自然用法可以保留。

**#6 判断标准**：翻译腔最爱滥用被动语态，直接 calque 英文的 "is being Xed" / "X was done to Y"。中文多数情况应该用主动主语或话题-评论结构。逐个检查「被」字句：

- **几乎总是翻译腔，必须改写**：「正在被 X」「在被 X」（正在被商品化、正在被压缩）；「被 + 持续/系统/整个/同时 + 动词」（被持续挤出去、被系统训练成、被整个删掉、被同时挤压）；「被 X 掉/被 X 出去」（被删掉了、被挤出去、被短路了、被拆了）。改写办法是换成主动主语或自然中文：执行正在被商品化 → 执行正在变成商品；环节正在被压缩 → 环节正在缩水；这个问题被删掉了 → 这个问题直接消失了；过滤器被拆了 → AI 把过滤器拆了。
- **可以保留的自然用法**：被替代、被贴上标签、被工具取代、被训练成、被要求、被骗。这些是中文里接受度高的「被」，描述主语真实地承受了一个不利或被指向的动作。
- **快速判断**：删掉「被」字，句子仍然通顺，说明这个「被」是多余的翻译味；出声念一遍，如果主语被某个抽象过程被动作用、读起来像翻译，就改成主动。

**特别注意**：修复「不是 X，是 Y」对照框架时，不要用「没有被……，它被……」这种「否定 + 被动」去替换——那等于用一个翻译腔换另一个，而且通常更糟（双重被动叠加先否定再肯定）。直接改成自然主动中文。

写完后还做一轮定性自查（对照验收标准的 8 条）。

## 常见失败模式

| 失败模式 | 症状 | 解法 |
|---------|------|------|
| 调研汇总而非作者写作 | 读完感觉"AI 搜了一圈写的摘要" | 没有形成 thesis。回去做视角匹配和叙事重构 |
| 围绕作者框架组织而非读者疑问 | 文章按"维度一+维度二"罗列，读者在某段想"关我屁事" | 重组为读者恐惧链/疑问链。每个案例先回答"为什么和读者相关" |
| 证据堆砌 | 多个章节证明同一个论点，读者在第一个案例后就想"所以呢" | 每个论点只用一个最强案例。thesis 前置：读者在前四分之一必须知道问题、判断和预期收获 |
| Relevance 不着地 | 读者读完前几段不知道"这和我有什么关系" | Opening 第二段必须把分析对象和读者的处境连接起来 |
| Opening 从技术概念切入 | 第一段就让非本领域的人皱眉 | 换成读者能直接感知的体感现象 |
| 时间维度模糊 | 同时暗示"现在重要"和"还很早期" | 时间维度判断必须在 opening 中正面表达 |
| Object-explainer | 始终从分析对象出发解释，不从读者处境出发判断 | 把主语从分析对象换成"你（读者）" |
| 内部框架泄露到成品 | 文中出现"从 L4 技术族谱视角"、axiom 编号 | 脚手架术语最终文章里零出现 |
| Cognitive burden 过高 | 连续多段引入新概念读者走神；或 taxonomy 宣布（第一/第二/第三）、机械承接词、信息墙把文章切成 checklist，读完像被灌了东西 | 每段新概念不超过两个；禁止 taxonomy 宣布、机械承接词、信息墙；数据嵌故事不罗列；出声念一遍检验节奏 |
| 破折号泛滥 | 全文大量使用 —— | 成品零容忍。写完用 `rg -n '——'` 一次性扫描 |
| 大段英文引文 | 中文文章里粘贴多句英文 blockquote | 改为中文重述，保留核心意思和 URL |
|「很+形容词」判断总结 | "展示得很直接""说得很清楚" | 用 `rg` 扫描，判断总结一律删除或替换为具体内容 |
| 元评论当过渡 | "具体来说""关键区别在于" | 直接进入内容。删除后文意不通说明前一句不够清晰 |
| 复杂比喻认知超载 | 一个段落塞入场景+问题+方案+洞察 | 一个比喻段只推一个概念，每段不超过 3 句 |
| 信息架构层面的认知超载 | 读者说"看不懂"或"只看懂了一小部分"，但逐句检查没有明显语法问题 | 根本原因不是句式而是信息密度：一个 section 塞了多个概念或多个案例。解法是砍内容，不是改写——一个 section 只保留一个核心判断 + 一个具体故事，其余删掉。参见"降认知负担的操作方法"章节 |
| sub-agent 重写变成同义词替换 | 第二遍语言层重写后逐段对比，大量原句雷同（超 30%） | `writer_deepseek` 必须按"从零新建"执行完全重写。主线程验收时逐段检查雷同率，超阈值要求该段或整篇推翻重来。若问题来自结构或信息密度，主线程先重组结构和删减内容，再重新交给 `writer_deepseek` |
| 无配图（分析/调研型） | 纯文字长文 | 数据图和流程图必须配，分析型文章配图可省 |
| 发布时间重复 | 正文写了日期，发布页又注入一次 | 正文不写发布时间 |
